# 客户四场景预置应用方案

## 目标

将客户提供的 4 个场景沉淀为 4 套独立的 `scene/` 预置应用，并作为后续需求澄清和生成能力画像选择的场景蓝本。第一版采用演示数据契约：应用以 mock provider 提供可替换的数据结构，保持静态 Vite 应用可构建、可部署、可演示，不直接接入真实潮汐、气象、AIS、Twitter 或 Instagram 服务。

## 已确认边界

- 每个场景对应一个独立 `scene/<slug>/` 项目，包含 `.factory/app.json`、README、Dockerfile、nginx 和自包含前端代码。
- 应用显示名保留客户场景名称；内部 slug、描述和场景蓝本名可用于区分同名场景。
- 应用内容保留客户判断口径，包括阈值、状态标签、场景解释和刷新频率。
- 门户应用卡片应显示一行副信息，优先取 manifest `description` 或 `slug`，用于区分两个同名客户场景。
- 应用界面显示客户原始刷新频率；本地演示可以使用较短 tick 推进“最近刷新时间”、倒计时或曲线变化。
- 4 个场景均归为 `指挥看板类应用`，生成能力画像使用 `software-factory-app`、`defense-operations-ui`、`command-dashboard`、`maritime-alert-dashboard`。

## 预置应用清单

| 场景 | 应用显示名 | 建议 slug | 内部区分描述 | 应用类型 |
|---|---|---|---|---|
| 场景一 | 航母母港潮汐窗口计算器 | `carrier-homeport-tide-window` | 四大航母母港潮汐窗口状态看板 | `command-dashboard` |
| 场景二 | 甲板风实时计算器 | `carrier-deck-wind-calculator` | 航母活动区域甲板风条件评估看板 | `command-dashboard` |
| 场景三 | 海域网格商船密度异常告警器 | `merchant-density-grid-alert` | AIS 商船密度网格异常告警 | `command-dashboard` |
| 场景四 | 海域网格商船密度异常告警器 | `social-sighting-cluster-alert` | 社媒海上目击聚合告警地图 | `command-dashboard` |

## 场景一：航母母港潮汐窗口计算器

### 客户口径

- 接入公开潮汐预测 API。
- 抓取诺福克、圣迭戈、布雷默顿、横须贺四大航母母港未来 72 小时潮汐数据。
- 航母吃水阈值设为 12.8 米。
- 自动标出各港口满足条件的“可出港时间窗”。
- 倒计时显示距离下一个窗口还有多久。
- 输出四格仪表盘，每格显示港口名、当前潮高、是否满足出港条件、下一个窗口起止时间。
- 窗口开放时显示绿色，关闭时显示红色倒计时。
- 每 10 分钟刷新一次。

### 第一版应用形态

- 主视图为 2x2 港口状态卡片。
- 每张卡展示当前潮高、阈值、条件状态、下一个窗口起止时间、倒计时、未来 72 小时潮汐曲线。
- 顶部显示数据源状态、最近刷新时间和“每 10 分钟刷新一次”。
- mock provider 输出未来 72 小时潮汐序列和已计算窗口，结构按未来潮汐 API 适配。

### 验收点

- 4 个港口必须同时可见。
- 绿色/红色状态必须与客户口径一致。
- 能展示窗口开放和关闭两种状态。
- 倒计时和最近刷新时间在本地演示 tick 下可变化。

## 场景二：甲板风实时计算器

### 客户口径

- 接入全球气象预报系统公开格点风场数据。
- 筛选全球所有已知航母活动区域的 10 米高度风速和风向。
- 舰载机起降所需甲板风最小值为 20 节。
- 结合航母可通过航向和航速自行合成甲板风的特性，用风速叠加航母最大航速 30 节，计算可实现的甲板风范围。
- 推算每艘航母当前是否具备“无弹射器辅助”或“安全着舰”的风力条件。
- 每 5 分钟更新。

### 第一版应用形态

- 主视图为全球航母活动区域列表 + 地图点位/海区面板。
- 每个活动区域展示 10 米风速、风向、航母最大航速 30 节、可实现甲板风范围、20 节阈值对比。
- 每艘航母展示“无弹射器辅助”和“安全着舰”条件状态。
- mock provider 输出格点风场摘要、航母活动区域、舰艇航速参数和评估结果，结构按未来 GFS 类数据适配。

### 验收点

- 20 节阈值和 30 节最大航速必须显式展示。
- 每艘航母必须同时展示两类风力条件判断。
- 至少包含满足、临界、不满足三类状态样例。
- 顶部显示“每 5 分钟更新”和最近更新时间。

## 场景三：海域网格商船密度异常告警器

### 客户口径

- 接入 AIS 船舶位置数据流。
- 将美在外活动航母区域按边长 50 海里划分网格监控。
- 统计各网格内正在活动的商船数量。
- 和过去 30 天同一海域平均商船密度对比，基准线取 30 天滑动平均。
- 商船数量突然锐减触发告警；数量正常或略高保持绿色。
- 每 3 分钟刷新一次。
- 当前数量低于基准线 70% 时黄灯，低于 50% 时红灯。
- 前端显示多个方格，每个方格内显示数量曲线。

### 第一版应用形态

- 主视图为海域网格矩阵或地图网格，每格代表 50 海里监控单元。
- 每格显示当前商船数量、30 天滑动平均基准线、当前占比、绿/黄/红状态、数量曲线。
- 右侧或底部显示告警列表，按红灯、黄灯、绿色排序。
- mock provider 输出网格定义、当前船舶数量、30 天基准线和历史曲线，结构按未来 AIS 聚合数据适配。

### 验收点

- 70% 黄灯和 50% 红灯规则必须显式展示并正确反映。
- 多个网格必须同时展示，并包含绿、黄、红状态样例。
- 每个网格内必须有数量曲线。
- 顶部显示“每 3 分钟刷新一次”和最近刷新时间。

## 场景四：海域网格商船密度异常告警器

### 客户口径

- 编写监控脚本，接入推特和 Instagram 的公开搜索接口。
- 监控关键词为“航母”“军舰”“大船”“海上”等，并用不同语言搜索。
- 限定地理范围为全球海域。
- 自动抓取含关键词的新帖子。
- 提取 GPS 标签或图片 EXIF 中的经纬度信息。
- 把所有带坐标的帖子落点在地图上形成散点图。
- 如果多个不同账号在同一海域短时间内发布类似内容，自动聚合并高亮。
- 提示可能是航母经过引发的目击潮。
- 每 15 分钟抓取一次新帖。

### 第一版应用形态

- 应用显示名仍使用客户原始名称。
- 主视图为全球海域散点地图 + 聚合高亮区域 + 新帖流。
- 每条帖子展示平台、账号、语言、关键词命中、坐标来源、时间和相似内容摘要。
- 聚合区域展示账号数、时间窗口、相似关键词、疑似目击潮提示。
- mock provider 输出社媒帖子、坐标来源、EXIF/GPS 标记、聚合结果和高亮区域，结构按未来社媒搜索/采集适配。

### 验收点

- 地图必须展示散点和聚合高亮两种层。
- 必须体现多语言关键词和 Twitter/Instagram 两类来源。
- 必须区分 GPS 标签坐标和图片 EXIF 坐标。
- 顶部显示“每 15 分钟抓取一次新帖”和最近抓取时间。

## 生成能力画像

4 个场景统一采用：

```json
{
  "base": ["software-factory-app"],
  "domain": ["defense-operations-ui"],
  "pattern": ["command-dashboard", "maritime-alert-dashboard"]
}
```

`maritime-alert-dashboard` 负责沉淀海域/港口/网格/坐标对象、外部数据演示契约、阈值告警、刷新节奏、倒计时窗口、地图网格、散点聚合等共性规则。

## 后续实施顺序

1. 补充 `maritime-alert-dashboard` 项目级生成能力包。
2. 更新门户应用卡片，显示 description 或 slug 作为副信息。
3. 逐个创建 4 个 `scene/` 预置应用，并生成 manifest。
4. 每个应用运行 `npm install`、`npm run build`。
5. 用 Factory 启动扫描确认 4 个新预置应用进入应用列表。
