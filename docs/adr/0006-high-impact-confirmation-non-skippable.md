# Never skip high-impact confirmation items during requirement clarification

The requirement-clarification adaptive flow must resolve every open 高影响确认事项 — any unresolved decision that can change business meaning, data source, external interface, permission, deployment, or user-visible behavior — before reaching `ready_to_confirm`, even when the user's first message is detailed. This reverses the earlier adaptive-method license that a sufficiently detailed first prompt could immediately enter `ready_to_confirm`: in practice the skill filled requirement fields from blueprint assumptions and treated them as confirmed, so a concrete first message produced the 确认并生成 button with no clarifying round at all, defeating the clarification gate. Non-high-impact details may still be assumed adaptively. The skill surfaces ALL open high-impact questions in a single round (batch) so the user confirms them together, rather than one per round — this updates the earlier one-question-per-round surface policy while keeping the gate itself unchanged; a field filled by assumption is explicitly not a confirmed high-impact decision. The server enforces the invariant so the 确认并生成 action cannot appear while a high-impact item remains open.
