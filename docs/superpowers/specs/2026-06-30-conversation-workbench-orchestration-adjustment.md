# Conversation Workbench Orchestration Adjustment

Date: 2026-06-30

## Status

Discovery complete; ready for implementation planning.

## Final Summary

The conversation workbench will show a leader-review layout with a fixed aggregate orchestration graph above the conversation area and the composer below. The graph contains `用户输入`, `业务逻辑`, `界面解析`, `数据抓取`, and `生产交付`; `协作编排` remains an internal planning capability in this iteration rather than a visible card.

The user-facing flow is serialized through one active clarification focus at a time: `业务逻辑` confirms the requirement summary before task creation and later displays the task-owned requirement document, `界面解析` produces the design document plus an interface preview that becomes the acceptance baseline after data compatibility checks, `数据抓取` verifies or degrades data sources and produces the data contract, then `生产交付` runs the downstream implementation, review, verification, build, and deployment sequence without routine user confirmation.

Each expanded clarification block shows a responsibility-specific progress track, live `思考过程`, `思考摘要`, clarification choices, artifact previews, and a confirmation action. Completed blocks fold into compact summary cards that retain artifact links and confirmation history. Session attachments are introduced as minimal composer uploads, previewable in the message timeline and later projected into the application workspace when relevant.

## Scope Notes

- This adjustment applies to the `sf-portal-mvp` conversation workbench and the Factory collaboration orchestration shown there.
- `sf-portal` is a demo project and is out of scope for this requirement.
- This iteration focuses on the conversation-workbench orchestration experience: four-card aggregate graph, serialized clarification focus, artifact confirmation, production-delivery aggregation, project-document preview, interface preview, and data-contract preview.
- This iteration does not implement the collaboration-agent library page, custom collaboration-agent CRUD, custom-agent direct launch, standalone sandbox runs, real dynamic summoning of custom agents, or `agents` category migration.

## Confirmed Decisions

1. The four visible orchestration cards are a user-facing display and aggregation layer, not a replacement for the internal collaboration-agent model.
2. Existing collaboration-agent keys, roles, machine execution contracts, audit records, and project document attribution remain stable.
3. The visible card labels are:
   - `业务逻辑`, backed by the current requirement-analysis responsibility.
   - `界面解析`, backed by the current interface-design responsibility.
   - `数据抓取`, backed by the current data-integration responsibility.
   - `生产交付`, aggregating the downstream delivery responsibilities that follow business logic, interface parsing, and data capture.
4. The conversation UI label is `思考过程`, not `原始思考过程`; after the live thought stream completes, it folds by default and shows a `思考摘要`.
5. The graph may show `界面解析` and `数据抓取` as the same dependency wave, but the conversation area has only one active clarification focus at a time: `业务逻辑` first, then `界面解析`, then `数据抓取`, then `生产交付` executes automatically.
6. After the user confirms the `业务逻辑` clarification, Factory creates the generation task. The task's requirement-analysis step then produces the machine output that is projected to `docs/01-requirements.md`; the Markdown requirement document is a task-owned project document, not a pre-task chat artifact.
7. After `界面解析` clarification is confirmed, Factory produces a task-owned interface preview artifact. It is usually a static preview, but may use a temporary preview deployment when dynamic interaction, routing, state, or mock runtime data is required. This preview is not a formal application version, application-store entry, or production delivery result.
8. After `数据抓取` clarification is confirmed, Factory shows a data-contract preview and projects the task output to `docs/data-integration.md`. The data-capture agent first verifies the ontology interface with user-provided base URL, OpenAPI address, authentication, request shape, and related details. If the ontology source is unavailable or unusable, the agent asks the user whether to degrade to public internet capture. If public internet capture is also unavailable or unusable, the agent asks whether to degrade to an explicit demonstration-data contract. The user may also request switching to ontology, internet capture, or demonstration data directly in the conversation.
9. Once the user confirms the interface preview artifact, it becomes the provisional interface acceptance baseline for production delivery and Factory retains a preview snapshot. After `数据抓取` confirms a compatible data contract, the same snapshot becomes the interface acceptance evidence for the production baseline. The delivered application should preserve the confirmed layout, component placement, core interactions, visual style, density, key copy, and field presentation. Allowed differences are limited to compatible real-data variation, responsive adaptation, loading/empty/error-state completion, engineering decomposition, performance fixes, and accessibility fixes. Significant differences, including data fields that invalidate the preview assumptions, require an interface-difference explanation in the conversation area and explicit user confirmation before continuing.
10. The `生产交付` card aggregates the internal downstream sequence: `方案设计 -> 代码生成 -> 代码审查 -> 测试验证 -> 产品验收 -> 镜像构建 -> 部署`; when security conditions apply, `安全审查` is inserted between code review and test verification. The card shows the current sub-stage, state, short summary, and progress trace, but does not ask for routine user confirmation. It pauses only for significant interface differences, data degradation, permission/deployment high-impact items, or failure repair decisions that require the user.
11. The `方案设计` sub-stage produces the task-owned machine output that is projected to `docs/02-solution.md`; this solution document is part of the production-delivery project documents.
12. `界面解析` confirmation covers an interface-parsing result package: `docs/03-design.md` plus the interface preview artifact. The document and preview must stay consistent. If they conflict, the user-confirmed preview is the acceptance baseline and the design contract/document must be updated to match it before production delivery continues.
13. Markdown document popups in the workbench are read-only rich previews. Users request changes through the conversation input; the owning clarification focus or task step regenerates or updates the machine output, project document, and downstream inputs.
14. Confirmation buttons approve the current card's artifact package, then Factory advances the flow. Suggested labels are `确认业务逻辑并继续`, `确认界面解析并继续`, and `确认数据抓取并继续`; the action is not a manual scheduler command.
15. When a clarification block is completed, its detailed conversation folds into a compact summary card that keeps the agent label, completion state, thinking summary, confirmed artifact links, confirmation time, and an expand action. Artifact links remain visible after folding.
16. The current leader-review layout shows only `用户输入` plus the four aggregate orchestration cards. It does not render `协作编排` as a separate card; orchestration may appear as graph title/status copy such as `编排已生成`. In this iteration, the visible four-card graph represents the user-facing default participation plan and is confirmed with the confirmed requirement summary; high-impact agent removal, custom participation editing, and separate plan-confirmation UI are deferred. Productized layouts should keep the graph renderer reusable so the same orchestration artifact can be fixed above the conversation or inserted as a conversation timeline item later.
17. A future `协作智能体库` should separate global definitions, per-dialogue/task participation plans, and task-local configuration snapshots. Built-in collaboration agents are viewable, enableable/disableable, and copyable, but not directly mutable; custom collaboration agents are created by copying or adding definitions and then selecting them into future plans. Running tasks always use frozen snapshots.
18. The existing `agents` table/API should become the shared resource model for agent definitions, but product categories should be made explicit before the management surface is expanded: `collaboration_builtin`, `collaboration_custom`, `business_processing`, and `managed_external`. This keeps collaboration agents separate from business-processing and externally managed agents while allowing the backend to reuse one definition store.
19. Custom collaboration-agent management is deferred. This workbench adjustment should not implement custom-agent creation, direct custom-agent launch, or standalone sandbox runs; it should preserve terminology and data-model compatibility for a later productization pass.
20. `业务逻辑`, `界面解析`, and `数据抓取` share one card-status set: `未开始`, `等待上游`, `待启动`, `执行中`, `等待用户澄清`, `产物待确认`, `已确认`, `失败`, and future-only `已跳过`. `执行中` cards use a breathing/pulse motion; `等待用户澄清` owns the conversation input; `产物待确认` exposes artifact links and the confirmation action. Card-specific artifact links differ: business logic exposes the confirmed requirement summary first and the task-owned requirement document after it exists, interface parsing exposes the design document and interface preview, and data capture exposes the data contract and sample-data preview.
21. This iteration includes a minimal session-attachment upload capability for requirement documents, API/interface documents, sample-data files, and screenshots. Uploaded files are stored as dialogue/session attachments associated with the current clarification focus; after a generation task exists, relevant attachments are copied, extracted, or summarized into the application workspace as controlled projected inputs. The iteration does not include complex attachment management, online editing, or long-term storage of plaintext credential files. Credentials, tokens, passwords, and API keys must use controlled credential input rather than normal attachment upload.
22. The attachment entry point lives in the conversation composer. Before send, selected attachments render as thumbnails or file chips inside the composer, each with a remove action. After send, the attachments render on the submitted conversation message as clickable attachment references for preview; they remain part of the message history and can later be projected into the application workspace when relevant.
23. Already-sent attachment references are not hard-deleted in the first version. A user may deactivate an attachment reference for future agent context, but the message-level reference remains visible for replay and audit. Hard deletion is deferred to a future attachment-management flow with explicit confirmation.
24. First-version attachment preview support is type-based: images show thumbnails and large preview; Markdown, text, JSON, and CSV show text preview, with CSV optionally table-rendered; PDFs open in an embedded preview or browser viewer; Word and Excel files show file metadata only and are excluded from automatic agent context extraction in this iteration; unknown types show filename, size, and type only and are excluded from agent context unless the user explicitly confirms inclusion. Rich Word/Excel extraction is deferred to a later attachment-management iteration.
25. A user may return to a confirmed upstream card, but doing so is an upstream-change rollback. Factory must explain the affected downstream artifacts before proceeding. If downstream steps have not started, the flow returns directly to the upstream clarification focus. If downstream artifacts were already confirmed, the user must confirm invalidating or revalidating them; previous artifacts remain in history and are marked as superseded by upstream change. If production delivery is running, Factory must pause or cancel the current production sub-stage, record the interruption reason, and then enter the upstream revalidation flow.
26. Production-delivery failures are handled by category. Repairable code generation, code review, and test-verification failures enter the bounded automatic repair loop and the production-delivery card shows `自动修复中`. Data-source unavailability, authentication failure, or fallback choice returns to the `数据抓取` focus for confirmation. Significant interface-preview deviation returns to the `界面解析` focus. Deployment permissions, port conflicts, external-service access, destructive operations, and repeated repair exhaustion pause in the production-delivery conversation block with a failure summary and user action choices. If the user abandons repair or rejects the required confirmation, the card transitions from `等待用户确认` to `失败` with the chosen reason recorded.
27. In the leader-review layout, the aggregate orchestration graph is fixed at the top of the conversation workbench content, not as a full-screen page header. When the conversation scrolls, the graph may collapse into a compact bar showing the current card, state, and progress copy such as `界面解析 · 等待用户澄清`; clicking the bar expands the full graph. Mobile defaults to the compact bar with an expandable horizontally scrollable graph.
28. On initial open with no active dialogue content, the conversation workbench shows the grey initial aggregate graph plus the conversation composer. `用户输入`, `业务逻辑`, `界面解析`, `数据抓取`, and `生产交付` are all `未开始`; after the user submits the first message, `用户输入` briefly enters the active breathing state and the flow advances to `业务逻辑`. No marketing empty state, task-creation interstitial, or visible `协作编排` card is shown.
29. The `业务逻辑` block displays the confirmed requirement summary directly before task creation. The user confirms this summary to create the generation task. After task creation, the requirement-analysis step projects `docs/01-requirements.md`, and the workbench adds it to the already-folded business-logic summary card as a read-only rich-preview document link. This is an incremental card update, not an automatic re-open. The requirement document must machine-validate against the confirmed summary. The requirement-analysis validator owns this consistency check, using a structured comparison such as field-set, summary checksum, or equivalent contract metadata. If they conflict, the mismatch is treated as a requirement-analysis `schema_validation_failed` failure, retried or repaired at the requirement-analysis step boundary within an explicit attempt limit, and never routed through the code-generation repair loop. The user-confirmed summary governs; this is not a second pre-task user confirmation gate.
30. Data-source fallback is confirmed step by step. The data-capture agent first attempts and reports ontology-interface verification. If ontology is unavailable or unusable, it asks whether to degrade to public internet capture. Only after the user confirms does it attempt internet capture and report the result. If internet capture also fails, it asks whether to degrade to an explicit demonstration-data contract. Each fallback changes the data-truth commitment and is treated as a high-impact confirmation; Factory must not silently skip directly to demonstration data. For customer-facing judgement applications covered by the real-data judgement constraint, demonstration data may support structure or preview only and must not be presented as a real judgement result.
31. If the user explicitly chooses the data boundary, data capture starts from that boundary rather than always trying ontology first. Choosing ontology starts ontology verification. Choosing public internet capture skips ontology and shows a data-truth notice that the result depends on public sources. Choosing demonstration/mock data skips real-source verification but must be explicitly marked as non-real data for demonstration or structural preview and treated as a high-impact confirmation. If the user does not specify a boundary, ontology remains the preferred starting point.
32. The `数据抓取` expanded conversation block includes a compact data-flow validation track with `来源 -> 处理 -> 流向`: source nodes for ontology, public internet capture, and demonstration data; processing nodes such as connection verification, sample acquisition, field recognition, and contract generation; and downstream nodes for data contract, interface compatibility check, and production delivery. Animation uses real validation states: active scan/flow while a step runs, solid flow on success, red breakpoints on failure, and waiting state for degradation confirmation. The top aggregate orchestration graph does not show these data details. After the data-capture block folds, the track becomes a mini data-link summary showing source path, fallback history, sample count, field count, and the data-contract link.
33. The `界面解析` expanded conversation block includes a lightweight interface-composition track. Inputs are the requirement summary, uploaded screenshots/prototypes where present, and any data fields already known from user input, attachments, or prior context; before `数据抓取` confirms the data contract, these fields are assumptions. Parsing steps include view recognition, layout partitioning, component mapping, interaction-state definition, and preview generation. Outputs are the design document, interface preview, provisional interface acceptance baseline, and production-delivery input. The track uses subtle step highlighting/scanning and preview-card activation, then folds into an interface summary with the design-document and preview links. When `数据抓取` later confirms a data contract, Factory runs an interface-data compatibility check. The minimum compatibility rule is that every field, state, or display value assumed by the confirmed preview must be provided by the data contract or have an explicitly confirmed fallback. If this check fails, Factory returns to `界面解析` for compatibility revalidation or explicit user confirmation.
34. The `业务逻辑` expanded conversation block includes a lightweight business-logic analysis track. Inputs are the user text, session attachments, and relevant historical context. Analysis steps include goal recognition, actor/object identification, business-rule extraction, clarification-need identification, and requirement-summary generation. Pre-task output is the confirmed requirement summary plus downstream interface/data constraints; after task creation, the requirement-analysis step projects the associated requirement document. The track uses subtle step highlighting and does not replace the thinking process, requirement summary, or later requirement document.
35. Expanded clarification blocks render in this order: agent title/status/current action, the responsibility-specific track, the live `思考过程` section, the `思考摘要`, replayable `模型分析过程` / `分析工作日志` where available, clarification questions or recommended choices, then artifact previews and the confirmation action. `思考过程` is expanded while running and folds by default after completion. Persisted replay uses the thinking summary and analysis work logs; raw thinking streams do not enter normal work-trace or project-document content. Pre-task clarification analysis logs are stored with the dialogue or clarification message history metadata, while task-stage analysis logs may also appear in work-trace records with task and step attribution; neither path stores raw thinking as ordinary work-trace or project-document content.
36. The existing `领域分析` collaboration agent is displayed inside the `业务逻辑` aggregate card for this leader-review layout. Its machine step, output, project document attribution, and audit attribution remain separate internally; only the user-facing orchestration graph aggregates it. Domain-analysis outputs update the folded business-logic summary card incrementally with links or trace summaries when available; they do not automatically reopen the card unless they need user input or invalidate the confirmed requirement summary.
37. `生产交付` has its own aggregate-card state set: `未开始`, `等待上游`, `待启动`, `执行中`, `自动修复中`, `等待用户确认`, `失败`, and `已交付`. The card always shows the current internal sub-stage and maps user waits to a reason such as interface difference, data fallback, deployment permission, destructive operation, or repair exhaustion.
38. When the user confirms an interface preview artifact, Factory must retain a preview snapshot before any temporary preview deployment is discarded. The snapshot may be a static build artifact, screenshot set, preview manifest, content hash, or equivalent immutable record sufficient for production-delivery comparison. While the baseline is provisional, the snapshot is a preview snapshot; after data-contract compatibility is confirmed, the same retained snapshot serves as the interface acceptance snapshot. The temporary preview runtime may be removed, but the retained snapshot remains.
39. External data access and attachment handling require an explicit safety boundary before implementation. Base URLs, OpenAPI URLs, and credential use must pass tool authorization; local, private-network, or otherwise sensitive fetch targets must not be contacted automatically. Credentials, API keys, tokens, passwords, and authorization headers use controlled credential input, are redacted before persistence or SSE, and are never written to project documents, ordinary work traces, or session attachments. Attachment upload must define size limits, allowed MIME/file extensions, storage ownership by `dialogue_id` and clarification focus, extraction-failure behavior, and retention/cleanup rules.
