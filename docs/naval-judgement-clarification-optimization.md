# 海军研判类应用需求澄清优化方案

## 目标

优化会话工作台中“新生成应用”的需求澄清能力，让 LLM 在面对海军、航母、海域、AIS、ADS-B、OSINT、潮汐、甲板风、商船密度、舰载机归属推断等需求时，能够先识别业务研判意图，再提出真正影响生成结果的领域关键问题。

用户侧的目标体验是：

- LLM 能明确说出它识别到的应用类型和关键约束，而不是泛泛地问“要做什么系统”。
- 澄清问题围绕海军研判业务边界，而不是普通 CRUD 应用字段。
- 对军方客户场景，不把 mock/demo 数据作为用户可选结果来源。
- 当前版本的数据来源澄清保持简单，只让用户多选“本体数据源”和“网络公开搜索”。

## 背景问题

客户给出的需求通常已经包含大量业务规则，例如：

- 航母母港潮汐窗口：72 小时潮汐预测、12.8 米吃水阈值、10 分钟刷新、四格仪表盘。
- 甲板风计算：10 米风场、20 节起降条件、航母最大 30 节合成甲板风。
- 商船密度异常：AIS、50 海里网格、30 天滑动均值、70%/50% 告警阈值。
- OSINT 目击聚合：公开搜索、GPS/EXIF 坐标、短时间多账号聚合。
- 舰载机归属推断：ADS-B、航母位置、海上起降、200 海里关联、60% 置信度。

如果澄清 prompt 只按通用应用字段提问，模型容易忽略这些研判规则，把关键边界埋进 `coreScenario` 或 `acceptanceFocus`。用户会感觉模型没有真正理解海军业务，也无法确认生成应用是否会按正确规则研判。

## 方案比较

### 方案 A：只改 prompt

只在需求澄清 skill 中加入“海军场景要问数据来源和研判规则”的说明。

优点：改动最小。

缺点：模型即使输出了研判边界，也没有稳定结构承载；前端确认摘要不会稳定展示；后端也无法持久化和测试。

结论：不推荐单独采用。

### 方案 B：新增轻量结构化研判边界

在现有澄清链路中新增 `judgementBoundary`，只包含：

- `dataSources`：数据来源边界，多选。
- `summary`：用户可读研判边界摘要。

同时更新 requirement-clarification skill、后端持久化、前端确认摘要和测试。

优点：范围小，兼容现有流程；能稳定展示和持久化；不会提前引入复杂数据源接入设计。

缺点：当前只解决“澄清与摘要”层面，不保证每一种外部数据源都已经有可运行 connector。

结论：推荐当前版本采用。

### 方案 C：一次性建设完整数据源接入框架

把本体、互联网搜索、Baidu、Twitter/X、Instagram、微博、AIS、ADS-B、潮汐、气象、海陆掩膜等全部建成可配置数据源，并在澄清阶段逐项确认接口、鉴权、字段、覆盖范围。

优点：执行边界最完整。

缺点：范围过大，会拖慢当前需求澄清优化；用户也已经确认当前先只做“本体数据源”和“网络公开搜索”两个来源族。

结论：作为后续“数据接入能力包 skills”方向，不放入本轮。

## 推荐方案

采用方案 B：新增轻量结构化研判边界。

本轮只优化澄清识别、关键问题、结构化摘要和前端确认展示，不扩展到具体数据源 connector 的执行实现。

## 适用范围

当用户请求包含以下意图时触发海军研判澄清规则：

- 航母、舰载机、军舰、舰船、海域、母港、潮汐、甲板风。
- AIS、ADS-B、商船密度、船舶位置、飞机轨迹。
- OSINT、公开搜索、目击聚合、GPS/EXIF、社媒线索。
- 归属推断、异常告警、态势复盘、事件关联、海上起降。

普通业务管理类应用不强制包含 `judgementBoundary`，避免把海军研判语言带入无关场景。

## 需求澄清策略

### 1. 先识别意图，再提问

模型在 `workLog` 中必须先说明识别到的场景类型和关键约束，例如：

> 识别到这是航母母港潮汐窗口计算器，关键约束包括 72 小时预测窗口、12.8 米吃水阈值、10 分钟刷新周期、四格仪表盘。

这个步骤的目的不是展示模型思考链路，而是给用户一个可验证的业务理解摘要。

### 2. 固定生成研判边界摘要

海军研判类需求使用统一摘要格式：

> 基于「xxx」数据，按照「xxx」规则，判断「xxx」。每「xxx」更新一次，以「xxx」形式输出。

示例：

> 基于「四大航母母港未来 72 小时潮汐预测」数据，按照「潮高满足 12.8 米吃水阈值」规则，判断「各港口是否满足出港条件及下一个可出港时间窗」。每「10 分钟」更新一次，以「四格仪表盘」形式输出。

### 3. 数据来源边界作为第一高影响问题

当前版本只问一个数据来源边界问题：

> 请选择本应用用于研判的真实数据来源（可多选）。

选项：

- 本体数据源
- 网络公开搜索

推荐默认值：

- 默认推荐“本体数据源”。
- 如果用户明确提到公开搜索、公开网页、社媒、OSINT、Twitter/X、Instagram、微博、目击线索，则推荐“本体数据源 + 网络公开搜索”。

### 4. 不问 mock / real

军方客户场景默认真实数据优先，`dataPolicy` 内部设为 `live_api`。用户澄清问题不出现“使用 mock 数据还是正式数据”的选项。

### 5. 不编造具体数据源

模型不能凭空说已经接入某个具体外部 API、平台或网站。

允许说：

- 客户已接入的真实本体数据。
- 公开网页、公开搜索或公开社媒线索。

不允许在未确认时说：

- 已接入某个具体潮汐 API。
- 已接入某个 AIS/ADS-B 数据库。
- 已接入 Twitter/X、Instagram、微博接口。
- 可以在运行时调用 Claude Code 搜索工具。

## 数据契约

在 `Requirement` 中新增：

```json
{
  "judgementBoundary": {
    "dataSources": ["ontology", "public_web_search"],
    "summary": "基于「...」数据，按照「...」规则，判断「...」。每「...」更新一次，以「...」形式输出。"
  }
}
```

字段说明：

- `ontology`：本体数据源。
- `public_web_search`：网络公开搜索。
- `summary`：确认需求摘要中展示给用户的研判边界。

`blueprintRefs` 仍然是内部 Factory 元数据，不能出现在用户侧事件或前端摘要中。

## 前端展示

确认需求摘要新增两行：

- 研判边界：展示 `judgementBoundary.summary`。
- 数据来源：把 enum 转成人可读中文标签。

展示示例：

```text
研判边界：基于「四大航母母港未来 72 小时潮汐预测」数据，按照「潮高满足 12.8 米吃水阈值」规则，判断「各港口是否满足出港条件及下一个可出港时间窗」。每「10 分钟」更新一次，以「四格仪表盘」形式输出。
数据来源：本体数据源、网络公开搜索
```

## 后端处理

后端需要支持以下问题 ID 合并到 requirement：

- `judgementBoundary.dataSources`
- `judgement_data_sources`
- `dataSources`
- `judgementBoundary.summary`
- `judgement_boundary_summary`
- `judgementBoundary`

模型输出的 `questions[].recommendation` 可以是数组，用于多选推荐。`openHighImpact[].recommendation` 的契约是字符串；后端对模型偶发输出数组做兼容，取第一个推荐值，避免整轮解析失败。

## 当前已验证行为

用“航母母港潮汐窗口计算器”场景进行本地 Claude Code dry-run，首轮结果满足核心预期：

- 识别为航母母港潮汐窗口计算器。
- 提取 72 小时、12.8 米阈值、10 分钟刷新、四格仪表盘等约束。
- `dataPolicy` 设为 `live_api`。
- 提出 `judgementBoundary.dataSources` 多选问题。
- 选项为“本体数据源”和“网络公开搜索”。
- 生成符合格式的 `judgementBoundary.summary`。

dry-run 也暴露了两个约束点，并已纳入方案：

- 要求 `multiSelect: true`。
- 禁止在未确认时举具体外部 API 例子。

## 测试策略

后端测试：

- `judgementBoundary` 可从结构化答案合并到 requirement。
- `judgementBoundary` 可从 consolidation 合并。
- 用户侧事件保留 `judgementBoundary`，但继续隐藏 `blueprintRefs`。
- `openHighImpact.recommendation` 兼容数组输出。

前端测试：

- 会话工作台 requirement summary 不丢失 `judgementBoundary`。
- 多选答案显示中文标签，而不是 JSON 数组或 enum。
- 独立澄清面板能接收并展示 summary/ready 事件中的 `judgementBoundary`。

已执行验证命令：

```bash
cd factory-server && go test ./internal/clarification ./internal/server
node sf-portal-mvp/scripts/check-dialogue-workbench.mjs
node sf-portal-mvp/scripts/check-clarification.mjs
```

## 不在本轮范围

本轮不处理以下事项：

- 不新增 Baidu、Twitter/X、Instagram、微博等具体搜索/社媒 connector。
- 不让生成应用在运行时调用 Claude Code 搜索工具。
- 不要求用户在澄清阶段提供本体实体、接口 URL、鉴权方式、字段映射。
- 不实现 AIS、ADS-B、潮汐、气象、海陆掩膜等数据源的完整接入能力。
- 不改变普通业务管理应用的澄清流程。

## 后续演进

后续可以通过“数据接入能力包 skills”扩展具体数据源能力：

- 本体数据源 skill：描述客户本体系统中真实存在的实体、接口、鉴权和字段。
- 网络公开搜索 skill：描述运行时可访问的公开搜索代理或检索接口。
- AIS/ADS-B skill：描述可用历史库、实时流、字段和覆盖范围。
- 潮汐/气象 skill：描述可用预测源、刷新频率和区域覆盖。
- 海陆掩膜/地理计算 skill：描述海上起降、距离计算、网格划分等基础能力。

这些能力包应该成为生成阶段可验证的执行边界，而不是在需求澄清 prompt 中自由编造。
