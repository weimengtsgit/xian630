门户把“人话”变成网络请求

这里的比喻是快递面单：用户写内容，前端贴上地址、格式和错误处理，再交给后端。

CODE 
const API_BASE_URL = import.meta.env.VITE_FACTORY_API_BASE_URL || 'http://127.0.0.1:8787'

async function request(path, options = {}) {
  const response = await fetch(`${API_BASE_URL}${path}`, {
    headers: { 'Content-Type': 'application/json', ...(options.headers || {}) },
    ...options,
  })
  if (!response.ok) {
    const body = await response.text()
    throw new Error(`${response.status} ${body}`)
  }
  return response.json()
}
PLAIN ENGLISH

默认把请求送到本机的工厂后端，也允许用环境变量换地址。

`request` 是统一寄件柜：所有前端请求都走这里。

`fetch` 是浏览器发网络请求的工具；这里会把 path 拼到后端地址后面。

请求默认声明自己发送的是 JSON，也允许调用方补充更多选项。

如果后端返回错误，代码先读出错误正文，再抛出一个可显示的错误。

如果成功，就把响应正文解析成 JavaScript 可以使用的数据。

实时状态同步

理解为什么任务进度和应用状态可以自动更新，而不用手动刷新页面。

门户像机场公告板

普通 REST 请求像去柜台问一次航班状态。SSE 更像机场公告板：一有变化，屏幕自动更新。

i
两种通信一起用

这个项目用 REST 拿完整列表，用 SSE 听变化通知。收到通知后，Hook 再刷新列表。

浏览器打开一条事件订阅

CODE 
export function subscribeFactoryEvents(onEvent) {
  const source = new EventSource(`${API_BASE_URL}/api/events`)
  const types = ['app.updated', 'job.created', 'job.updated', 'step.updated', 'artifact.created', 'deployment.updated']
  types.forEach(type => {
    source.addEventListener(type, event => {
      onEvent(type, JSON.parse(event.data))
    })
  })
  return () => source.close()
}
PLAIN ENGLISH

创建一个 EventSource，连接到后端 `/api/events`。

声明前端关心的事件类型：应用、任务、步骤、产物、部署。

每种事件都注册一个监听器。

事件到达时，把 JSON 字符串解析成对象，再交给调用方。

返回一个清理函数，组件卸载时关闭连接，避免留下旧订阅。

工厂会先安装依赖，再构建

依赖 安装策略不是随便选：有 lockfile 时用 `npm ci`，没有时用 `npm install`。这能兼顾可重复构建和新生成项目。

CODE 
	// 1. Dependency install: npm ci when a lockfile is present, else npm install.
	if _, err := os.Stat(filepath.Join(projectDir, "package-lock.json")); err == nil {
		if _, ok := f.runCmd(ctx, job, step, projectDir, "npm", "ci"); !ok {
			return StepResult{Status: model.StepStatusFailed, ErrorCode: model.ErrorDependencyInstallFailed, ErrorMessage: "npm ci failed"}, nil
		}
	} else {
		if _, ok := f.runCmd(ctx, job, step, projectDir, "npm", "install"); !ok {
			return StepResult{Status: model.StepStatusFailed, ErrorCode: model.ErrorDependencyInstallFailed, ErrorMessage: "npm install failed"}, nil
		}
	}
PLAIN ENGLISH

先看项目里有没有 `package-lock.json`。

有 lockfile 就用 `npm ci`，按锁定版本安装，结果更稳定。

如果安装失败，Step 失败并写入 `dependency_install_failed`。

没有 lockfile 就用 `npm install`，适合刚生成出来的项目。

这段代码把“怎么装依赖”固定在 Factory，而不是交给 AI 临场决定。

出错时看哪里

用 error_code、output.json、attempt artifact 和重试入口摆脱“AI 反复猜”的循环。

失败不是一句“坏了”

这个项目把失败当成黑匣子记录：每个阶段都要产出结构化 output.json，并把失败归到机器可读的 error_code。

M

output_missing

阶段没有写出 `output.json`。

J

output_invalid_json

文件存在，但不是后端能解析的 JSON。

B

build_failed

npm 构建失败，先看 stdout.log 和 stderr.log。

H

health_check_failed

容器启动后 HTTP 探活没有通过。

后端严格读取阶段结果

这里像海关查验文件：不是“看起来差不多”就放行，而是必须能严格解码。strict decode 会让格式漂移早点暴露。

CODE 
func readAndDecode(path string, target any) error {
	f, err := os.Open(path)
	if err != nil {
		if errors.Is(err, os.ErrNotExist) {
			return fmt.Errorf("%s: %w", path, ErrOutputMissing)
		}
		return fmt.Errorf("open %s: %w", path, err)
	}
	defer f.Close()
	dec := json.NewDecoder(f)
	dec.DisallowUnknownFields()
	if err := dec.Decode(target); err != nil {
		return fmt.Errorf("%s: %w", path, ErrOutputInvalidJSON)
	}
	return nil
}
PLAIN ENGLISH

打开指定路径的结果文件。

文件不存在时，包装成 `output_missing`。

确保文件最后会被关闭。

创建 JSON 解码器，并拒绝未知字段。

如果解析失败，包装成 `output_invalid_json`。

成功时返回 nil，表示这个基础契约通过。

排查顺序：先证据，后猜测

1
定位 Job 和当前 Step

确认状态是 running、failed、waiting_user 还是 canceled。

2
读取 error_code

先区分是 runner、schema、npm、Podman、端口还是健康检查。

3
打开 attempt artifact

查看 `output.json`、`stdout.log`、`stderr.log`，不要只看 UI 文案。

4
只重试当前阶段

失败后用 retry-current-step，避免整单重跑掩盖问题。
