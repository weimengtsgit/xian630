认识工厂角色

把 monorepo 里的服务、面板和智能体当成一组分工明确的角色。

这不是一坨代码，是一张分工表

monorepo 里有前端、后端、观测服务和多个场景应用。像剧组一样，导演、摄影、剪辑各做各的事，合起来才是一部片子。

UI

sf-portal

用户入口：显示应用、任务、智能体，并提交需求。

GO

factory-server

编排系统：排队、推进步骤、构建、部署、存状态。

CC

cc-status

观测系统：记录 Claude session、subagent 和 skill 状态。

APP

scene / generated-apps

可运行应用：预置示例和将来生成出来的新应用。

智能体也是注册出来的角色

右侧“智能体”面板不是从 UI 硬编码出来的。后端启动时把固定 registry 写入数据库，门户再读取。

CODE 
func DefaultRegistry() []model.Agent {
	return []model.Agent{
		{
			ID:              "agent_requirement_analyst",
			Key:             "requirement-analyst",
			Name:            "需求分析",
			Role:            "requirement_analysis",
			Description:     "把用户自然语言需求整理为生成应用的明确需求",
			ClaudeAgentName: "requirement-analyst",
			SkillsJSON:      "",
			Enabled:         true,
			SortOrder:       1,
		},
PLAIN ENGLISH

这个函数返回系统内置的一组智能体。

第一位角色的稳定 ID 是 `agent_requirement_analyst`。

`Key` 是机器用的名字，后续 Step 会用它匹配执行归属。

`Name` 和 `Description` 是给人看的中文说明。

`ClaudeAgentName` 预留给真正的 Claude CLI subagent 对接。

`Enabled` 和 `SortOrder` 决定这个角色是否启用，以及面板里的顺序。

组件聊天：谁找谁办事

门
sf-portal

我负责接收用户需求，也负责展示应用和任务状态。

厂
factory-server

把需求交给我。我会创建 Job，推进六个 Step，并记录结果。

观
cc-status

我不编排任务，只观察 Claude 的 session、subagent 和 skill 生命周期。

应
应用目录

我通过 `.factory/app.json` 告诉工厂自己怎么构建和运行。
